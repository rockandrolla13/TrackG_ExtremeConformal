# Historical Data Download Orchestration Design

**Date:** 2026-03-11
**Project:** TrackG_ExtremeConformal (CL-G4 Extreme Conformal Prediction)
**Purpose:** Multi-agent orchestration design for downloading FX and crypto historical data

---

## Overview

This document specifies how **orch** (multi-agent CLI orchestrator) will coordinate downloading free historical financial data for empirical experiments in extreme conformal prediction research.

### Objectives

1. Download 1-minute OHLCV data for 5 trading pairs:
   - **FX:** EUR/USD, GBP/USD, USD/JPY (2010-2024)
   - **Crypto:** BTC/USD, ETH/USD (2018-2024)

2. Validate data quality and presence of known extreme events

3. Generate reproducible download scripts and documentation

4. Execute end-to-end in headless mode (unattended)

### Approach

**6-step sequential pipeline** with fail-fast error handling:
- **Gemini** (1 step, 17%): Research data sources
- **Codex** (5 steps, 83%): Implement, execute, validate, document

**Execution model:** Headless autonomous pipeline that stops on first failure for manual intervention.

## Step-by-Step Specification

### Step 1: Research Data Sources [Gemini]

**Agent:** Gemini
**Timeout:** 600s (10 minutes)
**On failure:** stop

**Why Gemini:**
- 1M context window for reading long documentation
- Web search grounding for verifying live URLs
- Best at comparing multiple sources and trade-offs

**Input:**
- `data/DATA_DOWNLOAD_PROMPT.md` (requirements specification)

**Task:**
Research and evaluate data sources:
1. **HistData.com** - FX historical data (preferred for FX)
2. **Dukascopy** - FX historical data (backup)
3. **CryptoDataDownload** - Crypto historical data

For each source, document:
- Pairs available
- Date ranges
- Download method (direct link, form, API)
- File format (CSV, ZIP, compression)
- Authentication requirements
- Rate limits or known obstacles

**Output:**
Structured research report saved to `.orch/steps/step_01_*.md` containing:
- Recommended source per asset class (FX vs. crypto)
- Exact download URLs or procedures
- File format specifications
- Any gotchas (CAPTCHA, registration, etc.)

**Success Criteria:**
- All 5 required pairs confirmed available
- Download methods documented
- Clear recommendation provided

### Step 2: Write FX Download Script [Codex]

**Agent:** Codex
**Timeout:** 300s (5 minutes)
**On failure:** stop

**Input:**
- Research findings from Step 1
- `data/DATA_DOWNLOAD_PROMPT.md` (requirements)

**Task:**
Create `src/data/download_fx.py` that:
- Downloads EUR/USD, GBP/USD, USD/JPY
- Date range: 2010-01-01 to 2024-12-31
- Frequency: 1-minute OHLCV

**Requirements:**
- Dependencies: `requests`, `pandas`, `zipfile` only (no proprietary APIs)
- Handle rate limiting (sleep between requests)
- Resume capability if interrupted
- Progress logging
- Error handling for network failures
- Extract ZIP files if necessary

**Output:**
- `src/data/download_fx.py` (executable Python script)

**Success Criteria:**
- Script runs without syntax errors
- Handles compressed files if needed
- Resume logic works
- No API keys required

### Step 4: Execute Downloads [Codex]

- Do not proceed to validation if downloads incomplete

**Output:**

## Error Handling & Recovery

### Fail-Fast Strategy

When any step fails:
1. Pipeline immediately stops
2. Error logged to `.orch/steps/step_XX_*.md`
3. Desktop notification: "Step X failed: [reason]"
4. State saved to `.orch/state.json`

**Recovery procedure:**
```bash
# Review error
orch step status
cat .orch/steps/step_XX_*.md

# Fix the issue (update script, URL, etc.)
# Retry from failed step
orch step retry

# Or skip if optional
orch step skip
```

### Expected Failure Scenarios

**Step 1 (Research):**
- Data source URL changed → Manual fix: update URLs in prompt, retry
- Source requires auth → Manual decision: switch to alternate source
- Data not available → STOP: requirements cannot be met

**Step 2-3 (Script generation):**
- Syntax errors → Retry: Codex self-corrects
- Missing functionality → Manual: refine prompt, retry

**Step 4 (Download execution):**
- Network timeout → Retry: scripts have resume capability
- Rate limited → Manual: add delays to script, retry
- Disk full → Manual: free space, retry
- Source changed format → STOP: update scripts, retry from Step 2

**Step 5 (Validation):**
- Extreme events missing → STOP: investigate data quality
- Large gaps → Manual decision: acceptable or re-download?
- File corruption → Retry Step 4 (re-download)

**Step 6 (Documentation):**
- Unlikely to fail (simple file writing)

### Resume Capability

Pipeline can be resumed after:
- Manual interruption (Ctrl+C)
- System reboot
- Network failure during downloads

```bash
# Resume from where it left off
orch resume --headless
```

## Rationale & Design Decisions

### Why Sequential (Not Parallel)?

- Research must complete before script generation
- FX and crypto scripts could be parallel, but minimal time savings (~2 min)
- Sequential ensures clean failure isolation
- Downloads must be sequential (network bandwidth constraint)

### Why Gemini for Research Only?

- Gemini's strengths (1M context, web search) best utilized for research
- Codex superior for code generation
- 17% Gemini / 83% Codex aligns with task nature (mostly code)

### Why Fail-Fast (Not Retry)?

- Data sources can have structural changes (not just transient failures)
- Scripts need human review if sources change
- Aggressive retry masks real problems
- Manual intervention ensures data quality

### Why Headless Mode?

- Downloads take 30-60 minutes (requires unattended operation)
- All steps have clear success criteria (don't need human judgment)
- Logs capture everything for post-hoc review
- Desktop notifications alert on completion/failure

### Why Separate Scripts (Not Monolithic)?

- Independent testing (FX vs. crypto)
- Easier debugging (isolate which source failed)
- Reusable (can run FX-only updates later)
- Clearer code organization
